在大模型应用越来越复杂的今天，上下文窗口的限制成为制约 Agent 能力的关键瓶颈。本文整理了各家 AI 产品在上下文压缩方面的工程实践，从方法论到实现细节，希望能给正在构建 Agent 系统的开发者一些启发。

## 为什么上下文不是越长越好

上下文压缩要解决的核心问题是：**在有限的注意力预算内，让模型看到最关键的信息**。

Agent 是自主执行的，一个复杂任务跑下来，可能要几十次工具调用。每次工具结果少则几百 token、多则几万 token——一个文件读取就是几千 token，一个 shell 命令的输出可能上万。

50 次工具调用，每次平均 2000 token，光工具结果就 10 万 token 了。再加上系统提示词、对话历史、用户指令……200K 的窗口很快就不够用。

更现实的问题是**成本**。即使模型支持 1M 上下文，每次推理都喂 1M 进去，token 费用是普通对话的几十倍。不做压缩的话，烧钱速度能让你心跳加速。

所以上下文压缩不是"有了就更好"的优化，而是"不做就跑不动"的基础设施。

## 各家方案对比

| 产品/方案 | 核心方法 |
|-----------|----------|
| **Manus** | Reduce（缩减）、Offload（卸载）、Isolate（隔离）三板斧 |
| **Claude Code** | 三层防线：Tool Result Clearing → Compaction → 子 Agent |
| **OpenClaw** | ContextEngine 接口 + head+tail 截断策略 |
| **Cursor** | Dynamic Context Discovery，按需加载减少 46.9% token |
| **LangChain** | 自主压缩工具，Agent 自己判断压缩时机 |

## Manus 的方法论

Manus 的上下文工程做得最体系化，把上下文工程拆成了三板斧。

### 缩减：两阶段逐级加码

**第一阶段是 Compaction**。Manus 给每个工具调用维护了两个版本：**完整版**和**紧凑版**。完整版就是原始工具输出，存在 sandbox 文件系统里；紧凑版只保留一个引用，比如文件路径。

随着对话推进，老的工具结果从完整版切换到紧凑版，新的保持完整。这样模型做下一步决策的时候，最近的信息是全的，老的信息只留个指针。

这里有个关键的设计理念——**所有压缩都是可恢复的**。即使网页内容移出上下文了，URL 还在，拉回来就行。同样，即使文档的内容被去掉了，但 sandbox 里还有文件路径。信息没有真正丢掉，只是暂时不在模型的"视野"里。

**第二阶段是 Summarization**。当大部分老结果都 compact 完了，窗口还是太大，就切换到轨迹摘要。但 Manus 不是让模型自由发挥写个总结，而是用**预定义的 schema**来约束摘要格式。

说白了，就是在做 summary 的时候，不要让模型自由发挥，而是给他一个表格一样的模板，让模型能一个个信息往里面填，这样 summary 的质量和可靠性会高不少。

### 卸载：把文件系统当成"外挂大脑"

上下文窗口有 token 限制、有性能衰减、费钱。但文件系统呢？大小不受限，天然持久化，Agent 可以直接读写。

**Manus 让模型学会了把文件系统当外部记忆用**。信息用完写到文件里，需要的时候再读回来。

工具层面也是这个思路。Manus 只有**不到 20 个**原子化工具函数——Bash、文件系统、代码执行这些基础的。其他功能不绑定成 LLM 的 tool，而是**通过 CLI 暴露在 sandbox 里**，Agent 用 Bash 去调用。

这样做的好处太直接了：工具定义在上下文里占的空间极小，模型选工具也更准——选项少，不容易选错。

### 隔离：多 Agent 不是为了分工，是为了分上下文

Manus 的多 Agent 架构有一个特别清醒的认知：

> "人类按角色组织是因为认知限制。LLM 不一定有这些限制。"

**子 Agent 的主要目标不是按角色分工（设计师、工程师、PM 那套），而是隔离上下文**。

架构上分三层：Planner 分配任务，Knowledge Manager 决定什么信息要持久化到文件系统，Executor 干具体的活。

任务委派有两种模式：
- 简单任务只传指令，子 Agent 在干净的上下文里独立工作
- 复杂任务共享完整上下文，但子 Agent 仍有独立的工具集

两种模式下都用**约束解码**确保输出格式严格可控。

### Manus 踩出来的几个坑

**别在系统提示开头放时间戳**。尤其是精确到秒的那种。看起来是个小事，但会把 KV Cache 命中率直接干到地板。

**保留错误，别擦除**。Agent 犯错了，本能反应是清理痕迹、重试。但 Manus 发现，擦除错误就是移除证据。没有证据模型就无法适应。把失败的尝试留在上下文里，模型看到报错和堆栈跟踪，反而会降低重犯的概率。

**小心 few-shot 陷阱**。Agent 重复处理类似任务（比如审查 20 份简历），上下文里全是相似的操作序列，模型会陷入"节奏"，开始机械重复。Manus 的解法是引入结构化的随机性——不同的序列化模板、替代措辞、格式上的微小变化。

## Claude Code 的三层防线

Claude Code 的方案可能是目前跑得最成熟的。

**第一层：Tool Result Clearing**。最轻量的手段——对话深处的工具调用结果，模型不再需要了，直接清掉。几乎没有信息损失，因为那些老结果早就被消化过了。这是"成本最低、风险最低"的压缩操作。

**第二层：Compaction**。对话快到上限了，让模型读一遍完整历史，生成结构化摘要。这个摘要会保留任务进度（"5/17 个文件改完"）、当前在做什么、已做的决策和原因、未解决的问题。压缩完带着摘要 + 最近 5 个文件继续干活。

**第三层：子 Agent**。复杂任务拆给子 Agent，各自在干净上下文里工作。探索了几万 token 的内容，只返回 1-2K 的摘要。10-20 倍的压缩比。

这三层的优先级是递进的——能用轻量手段解决就不上重量级的。

## OpenClaw 的工程细节

OpenClaw 是开源的，所以你能直接看到代码层面的实现。

它设计了一个叫**ContextEngine**的可插拔接口——ingest（摄入）、assemble（组装）、compact（压缩），整个上下文生命周期管理得很清晰。

工具结果截断有个巧妙的设计：如果尾部有错误信息或 JSON 结构，用**head+tail**策略截断，中间省略但保留首尾。shell 命令的报错通常在最后面，直接截尾部就丢了最关键的信息。

分段摘要也做得很细。消息按 token 份额切段，分段生成摘要再合并。有 fallback：某条消息太大导致摘要失败，跳过它标个 `[Large message omitted]`，不至于整个流程挂掉。

几个关键参数：
- 单个工具结果不能超过上下文窗口的 **30%**
- token 估算加 **20% 安全边距**，补偿中文等多字节字符的低估
- 保留最近三轮对话保持原样、不压缩
- 给下一轮对话至少预留 20K token 的空间

摘要指令里强制要求 "Preserve all opaque identifiers exactly as written"，防止 UUID、文件路径被模型"简化"。

## Cursor 和 LangChain 的思路

**Cursor**的 Dynamic Context Discovery——与其压缩，不如一开始就少放东西。工具描述按需加载，大输出写文件，Agent 用 `grep`/`tail` 按需读取。A/B 测试 token 消耗减少了 **46.9%**，而且模型表现反而更好了。

**LangChain**的 Deep Agents SDK 加了一个"自主压缩工具"——不是固定阈值触发，而是 Agent 自己判断"现在是个好时机"然后触发压缩。

LangChain 的核心假设是：Agent 比固定规则更懂"现在是不是个好时机"。比如：
- 刚完成一个子任务的时候（中间状态已经转化为结果了）
- 从大量数据里提取完结论之后（原始数据不再需要）
- 准备开始消费大量新信息之前（腾出空间）

## 我的理解

研究完这一圈，我觉得有几个认知层面的东西比具体技术方案更重要。

**第一，上下文压缩的本质是注意力管理。**不是在做数据压缩，是在帮模型"聚焦"。你不是在尽量多地保留信息，而是在确保**模型当前决策所需要的信息**在它的注意力范围内。

**第二，压缩和恢复必须是一体的。**Manus 的"可恢复压缩"、Claude Code 的"Tool Result Clearing + 保留文件引用"、OpenClaw 的"占位符替换"，本质上都是同一个思路：**压缩不等于删除，只是暂时移出视野**。

**第三，KV Cache 是被严重低估的优化维度。**10 倍的成本差异，不是开玩笑的。但很多人在设计 Agent 的时候根本没想过这件事——系统提示里放个时间戳、动态增删工具、JSON 序列化不保证键顺序——随便一个就能让缓存全部失效。

**第四，没有银弹，但有清晰的优先级。**从各家的实践来看，上下文管理基本是这个优先级：

1. **先管住入口**：不必要的信息不要放进来
2. **再管住工具结果**：这是最大的 token 消耗源
3. **然后做对话压缩**：Compaction / Summarization
4. **需要的话拆 Agent**：上下文隔离
5. **兜底靠文件系统**：外部记忆

从轻到重，能用简单手段解决的就不上复杂方案。

## 最后

上下文工程这个领域，说实话还挺早期的。各家都在摸索，没有谁的方案是"标准答案"。

但方向已经比较清晰了。Andrej Karpathy 把它定义为：

> "context engineering is the delicate art and science of filling the context window"

即"填充上下文窗口的精妙艺术和科学"。我觉得说得挺对——这确实既是工程问题（大量的参数调优和边界处理），也是一门艺术（什么该保留、什么该丢弃，很多时候是个判断题，不是计算题）。

如果你在做 Agent 相关的东西，上下文管理这块真的值得花时间深入研究。
